## 1. What Carry Is

Carry is a native iOS app for tracking **skins games** — a common format where golfers in a group compete hole-by-hole for a shared pot. The app replaces paper scorecards and disputed post-round calculations with real-time live scoring, automatic payout math, and a shared scorecard across multiple tee times.

Today, Carry supports:

- **Quick Games** — one-time skins games between friends
- **Skins Groups** — recurring groups that play together over a season (tracks season-long stats and leaderboards)
- **Multi-group rounds** — up to 5 tee-time groups (20 players total) competing in a single round, with scores consolidated across all groups in real time
- **Net and gross scoring** — players compete on a level playing field using course-handicap-adjusted net scores
- **Handicap stroke allocation per hole** — strokes distributed by hole handicap index per USGA rules (including plus handicaps)

## 2. Who Uses Carry

- Club members at private and public golf clubs running weekly or monthly skins groups
- Friends organizing one-off competitive rounds
- Golf travel groups, buddy trips, member-guest weekends
- Handicapped golfers who want equitable competition with players of different skill levels

Carry is a consumer product priced as a low-cost subscription. Users are golfers, not commercial entities, and the data they provide is used solely to enable their own group's competition.

## 3. The Problem GHIN Data Solves

Carry's core value depends on **accurate, up-to-date handicap indices**. Today, players self-enter their handicap during onboarding, which produces three failure modes:

1. **Stale handicaps.** A player's handicap changes after each posted round under the World Handicap System. Manually-entered handicaps drift out of date, giving some players an unfair advantage or disadvantage.
2. **Dispute resolution.** In real-money skins games, an incorrect handicap is a source of friction. "What's your actual index?" conversations are common.
3. **Plus handicap and edge cases.** Players often mis-enter plus handicaps, or enter a course handicap instead of a handicap index. Server-authoritative data eliminates the error.

GHIN API access solves all three by letting Carry fetch the player's current, verified Handicap Index directly from the USGA's system of record.

## 4. Specifically How Carry Will Use GHIN Data

### 4.1 Handicap Index (primary use case)

- **When:** On user onboarding, and on manual refresh from the user's profile.
- **What:** The user's current Handicap Index.
- **Why:** To populate the user's profile and drive net-score calculations for their skins games.
- **User control:** The user enters their GHIN number and authorizes the lookup. They can remove the GHIN number from their profile at any time, which severs the link.

### 4.2 Course / Slope Rating (secondary)

- **When:** When a user selects a course and tee box for a round.
- **What:** Course rating, slope rating, hole-by-hole handicap indexes.
- **Why:** To compute the correct course handicap (`Handicap Index × (Slope / 113) + (Rating − Par)`) and allocate strokes per hole under USGA rules.
- **Alternative:** Carry currently uses a secondary golf-course data provider for this. GHIN's course data would be used as the authoritative source where available.

### 4.3 Golfer Profile Verification

- **When:** When a user enters a GHIN number during onboarding or profile editing.
- **What:** Confirmation that the GHIN number exists and belongs to the authenticating user (via name / club match).
- **Why:** To prevent accidental typos and unauthorized use of another golfer's GHIN number in competitive contexts.

### 4.4 Score Posting (future, user-initiated only)

- **When:** After a round completes, if the user explicitly chooses to post their score to GHIN.
- **What:** Adjusted Gross Score, date, course, tee, slope/rating.
- **Why:** So players don't have to re-enter the round into the GHIN app separately. Every post would be an explicit user action — no background posting, no bulk operations.
- **Status:** Not implemented in initial release. Scoped for post-launch once API access is approved.

### 4.5 Score History (not planned)

Carry has no plan to pull a user's score history from GHIN. Carry's own score data is generated by the app's scoring flow, and there is no product need for historical GHIN scores.

## 5. What Carry Will NOT Do

- **No commercial resale** of GHIN data or derived data
- **No scraping or spidering** — only authorized OAuth API calls
- **No credential sharing** between users
- **No public display** of a user's handicap or scores beyond their own opted-in groups
- **No data harvesting** — Carry retrieves the minimum data required for each user-initiated action
- **No caching beyond session** — Handicap Index is cached locally for offline use during a round, but refreshed from GHIN on next online session
- **No access to other golfers' GHIN data** except via the app's normal group flow, where players are invited and consent to sharing their own handicap with the group

## 6. Technical Implementation

- **Authentication:** OAuth 2.0 with the USGA-issued client credentials, stored server-side in a Supabase edge function. The iOS client does not hold OAuth secrets.
- **Rate limiting:** All GHIN API calls routed through a server-side edge function, allowing us to respect rate limits, cache where appropriate, and centrally manage compliance.
- **Logging:** API calls logged for audit and debugging; logs retained 30 days and access-controlled.
- **Error handling:** If GHIN is unavailable, the app falls back to the user's last-known Handicap Index or manual entry. GHIN is an enhancement, not a single point of failure.
