## BUG 1 FIX — canonical-identity design (Option B) + stage tracker

### Why the id splits (confirmed against foundation code)
`create_guest_profiles` (`20260328000000_guest_profiles.sql`) does `new_id := gen_random_uuid()`
SERVER-side — the client cannot supply the UUID. So a guest's `Player.id`
(`= stableId(from: uuid)`) **necessarily changes** at the local→server transition:
`stableId(localUUID)` → `stableId(serverUUID)`. Two devices mint two server UUIDs for one
human → two ids. Any dedup on the raw Int id lets both through.

### Canonical model (the fix)
1. **One key for all dedup/membership/merge:** `Player.canonicalKey`
   = `profileId` if set, else `inviteMemberId` if set, else the local int id.
   (QuickStart already derives ids from UUIDs correctly — this generalizes it.)
2. **All guest births UUID-derived, never a counter.** Eliminate `nextGuestID`.
3. **Heal the id at the server transition:** when a guest gains a `profileId`, set
   `id = stableId(profileId)` AND rewrite the snapshot (so old id is replaced, not doubled).
4. **Snapshot canonicalization:** `GuestSnapshot.asPlayer` derives id from profileId when
   present; `save()` dedups by canonicalKey. Heals already-poisoned `guest_roster_json`.
5. **Cross-device same-name collapse — DEFERRED to a separate decision (the judgment call).**
   `guestHumanKey` (name+handicap) collapse would fix the cross-device case (two devices each
   mint a server UUID for one human → two profileIds → canonicalKey can't collapse them). BUT
   it risks merging two LEGITIMATELY different guests who share name+handicap (e.g. two "Mike"s,
   both handicap 0.0 default) → under-counts the pot = wrong money the OTHER direction. So it is
   NOT auto-applied. Rationale it's likely unnecessary for Daniel's actual bug: the save-path
   deduped by Int id (S4), so the within-device divergence (one guest as both local-id AND
   server-id) almost certainly produced the dupes he saw — which canonicalKey + remap fixes
   cleanly and safely. Cross-device name-collapse will be offered as an explicit opt-in after
   the safe fix lands. `guestHumanKey` stays in Player.swift unused-by-default for that purpose.
6. **Money safety net:** dedup-by-profileId before building `playerTuples` in
   `RoundCoordinatorView.createSupabaseRound`.

### Invariant impact (guest-lifecycle locked rules — all must still hold)
| Invariant | Touched? | How preserved |
|---|---|---|
| Guests in `round_players` only, never `group_members` | No | identity-only change; no table writes added |
| Round-start reconciliation recreates missing profiles | Yes (improve) | ADD id-remap after profileId assigned (step 3); names still from snapshot |
| Disease string "Guest"+0.0 defenses | No | keep all 4 filters; canonicalKey change is orthogonal |
| 4-layer guest-edit persistence chain | Yes (careful) | dedup-by-canonicalKey must not drop the edited copy; reconcile by profileId |
| Games-tab pills 1:1 with roster | Yes (improve) | collapsing dupes makes pills correct, not lossy |

## BUG 2 — Score sheet Save/Score-Next buttons cut off

### Root cause
The score drawer height is **fixed** at `UIScreen.main.bounds.height * 0.62` and the
content is wrapped in `.clipped()` (ScorecardView). The dial + header + buttons need a
fixed **~522pt**. On the ~844pt reference device `0.62 × 844 ≈ 523pt` — *just* fits, so it
passed dev testing. On any **shorter** phone `0.62 × H` drops **below** 522pt and
`.clipped()` silently severs the bottom (the Save / Score Next buttons).
→ **Device-height-dependent. That's why it shipped undetected.**

### Fix (design-preserving, per Daniel: "rewrite without changing the design at all")
Height becomes `max(0.62×H , intrinsicContentHeight + 36)`:
- Reference device: `0.62` term still wins → **pixel-identical**, nothing moves.
- Shorter devices: content floor wins → buttons (already part of the design) stop clipping.
- Device-independent → **Tyler's phone model no longer needed.**

### Files changed (uncommitted on `hotfix/1.1.2`)
| File | Change |
|---|---|
| `Carry/Views/ScoreInputSheet.swift` | + `static func minimumContentHeight(extraBottomPadding:)` — sums the fixed layout blocks (header 44+16, 2× spacer 24, dial 96×3, buttons 18×2+24, bottom 30+extra). Fonts are fixed-size (no Dynamic Type here) so it's exact per device. |
| `Carry/Views/ScorecardView.swift` | drawer `.frame(height:)` → `max(0.62×H + collapsed, minimumContentHeight(...) + 36)`. Comment explains the reference-device equivalence. |

## BUG 1 — Duplicate guest users

### Symptom (Daniel)
Guests appeared **twice** when assembling tee groups — on his device AND the other scorer's.
Restarted the round, cleaned the dupes, **they came back**. Real Carry user (Keith, the
scorer) was NOT duplicated; only guests.

### Root cause — guests have TWO identities that never reconcile
`Player.id` is an `Int`. For a guest it is set two different ways:
- **Local create** (`addGuest`, PlayerGroupsSheet) → a local counter `nextGuestID` (100, 101, 102…).
- **Server load-back** (from `round_players`) → `Player.stableId(from: profileId)` = a hash of the UUID.

So the same person becomes `Kyle@100` (local) **and** `Kyle@<bighash>` (server-loaded).
Every roster-merge / dedup keys on `Player.id`:
- `QuickGameGuestStorage.save`: `seen.insert($0.id)` (Int)
- `GroupService` between-rounds backfill: `existingIds = Set(players.map(\.id))` (Int)
- `GroupManagerView` preservedGuests / source-4 merge: `freshIds`/`memberIds` (Int)

`100 ≠ bighash` → dedup misses → **both pills render.**
**Keith** is a real Carry user — always loaded from the server with the *same* id both
ways → dedups → no dupe. That's the tell: only guests double.

### Why it survives Restart Round (self-reinforcing loop)
The merge writes **both** copies back into `skins_groups.guest_roster_json` (durable
snapshot). Restart Round wipes the guest *profiles* but then **reads the poisoned
snapshot** → recreates two → dupes return. Same *shape* as the documented "Guest"+0.0
disease cycle (see `guest-lifecycle` skill, Disease-string rule).

### What still needs confirming before coding
Whether there are *also* genuinely duplicate **server profiles** (each scorer device minted
its own Kyle via `createGuestProfiles`) layered on top of the client loop.
Can't query prod directly (RLS-gated anon key; no service-role key locally).
→ Diagnostic: `docs/investigations/franks-payday-dupe-diagnostic.sql` (100% read-only).
Run **Query D** in Studio (project `seeitehizboxjbnccnyd` = Carry/Ohio = LIVE db).
- `distinct_player_ids = 1` per name → pure client/snapshot loop.
- `distinct_player_ids = 2` → server profiles also duplicated (fix needs server cleanup too).

### Proposed long-term fix (NOT yet implemented — Daniel: "best long-term fix")
Direction (pending Query D result + permission per "no patches" rule):
1. **One canonical identity:** guests keyed by `stableId(profileId)` everywhere; route ALL
   guest creation through `createGuestProfiles` so a server UUID exists *before* the Player
   gets an id (kills the local-counter divergence at the source).
2. **Dedup on `profileId`** (+ name/handicap safety key) across all 4 roster sources + the
   snapshot save path — not on the unstable Int id.
3. **Heal the snapshot on load:** collapse existing `guest_roster_json` dupes so already-poisoned
   groups self-clean.
Must preserve every guest-lifecycle locked invariant (guests in `round_players` only;
4-layer persistence chain; pills 1:1 contract; disease-string defenses).

## ROOT CAUSE (one cause behind every guest bug)
A guest's identity CHANGES during its life:
- Born: local UUID → `id = stableId(localUUID)`, `profileId = nil`.
- `create_guest_profiles` mints a FRESH server UUID → guest rebuilt with a DIFFERENT id + profileId.
- Restart: `delete_quick_game_guests` wipes the profile → reconciliation mints YET ANOTHER UUID → id flips again.
Dups = two representations of one guest surviving a merge. Vanishing = id remap desyncs from `playerIDs`.

## DECISION: stable-UUID architecture (pending Daniel's final GO)
Give each guest ONE UUID at birth, persist it, reuse it FOREVER — including supplying it to
`create_guest_profiles` (small dev-DB migration to accept a client-provided id) and reusing it on every
restart. Then `id` never changes → no remap (delete line 576) → no two-representations → DELETE the
canonicalKey / dedupedByCanonicalKey / recentlyRemovedIds-as-String scaffolding. One identity, cradle to
grave. RETIRES the bug class instead of tracking it.

Scope honesty:
- Kills the WITHIN-device instability permanently (born-local → server-remint → restart-remint). Dominant cause.
- Does NOT alone kill genuine CROSS-DEVICE same-name (two devices each create "Kyle" → two real UUIDs).

## INVESTIGATION CONFIRMED (2026-05-30) — exact duplication mechanism

**The mechanism, step-by-step (Restart Round path, which is what Daniel hit):**

1. Daniel creates QG: `handleQuickGameCreate` calls `createGuestProfiles` → server `gen_random_uuid()`
   mints UUIDs (call them `UUID_A`). Guest Players: `id = stableId(UUID_A)`, `profileId = UUID_A`.
   Snapshot saved with `UUID_A`. allMembers/server agree. ✅
2. Daniel taps Start Round, sees "only me on scorecard" (the vanish bug — likely a timing race
   in my reconciliation block, separate but related; needs repro).
3. Daniel taps Restart Round → `RoundService.deleteRound(roundId)` runs →
   `delete_quick_game_guests(roundId)` wipes the guest profile rows server-side (the UUID_A
   profiles are GONE from `profiles` table).
4. Daniel taps Start Round again → `createSupabaseRound` runs reconciliation
