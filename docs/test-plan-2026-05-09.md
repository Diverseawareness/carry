# Test plan — 2026-05-09 (1.0.7 hotfix + push-trigger Vault hardening)

Two bundles shipped today. This plan validates both before any App Store / TestFlight cut.

## What changed

### 1. Server: push-trigger auth moved to Supabase Vault
Migration: [supabase/migrations/20260509000000_notify_push_use_vault.sql](../supabase/migrations/20260509000000_notify_push_use_vault.sql)

- All four push-firing functions (`notify_push`, `send_handicap_reminders`, `reconcile_phone_invites_for_profile`, `reconcile_phone_invite_at_insert`) now call shared helpers `_push_notification_url()` + `_push_notification_anon_key()` reading from Vault.
- Helper chain is Vault → GUC → empty fallback (safe for environments without Vault populated).
- Per-environment Vault secrets inserted via `vault.create_secret()`: `supabase_anon_key`, `supabase_push_url`.
- "Verify JWT with legacy secret" toggled back **ON** for `send-push-notification` on both prod (`seeitehizboxjbnccnyd`) and dev (`gbhljwtbobbxervekxkg`). The function URL is no longer publicly callable.
- **Status:** APPLIED to dev + prod 2026-05-09. No client change required — every shipped 1.0.x build benefits immediately.

Background: `app.settings.supabase_anon_key` GUC was NULL on both projects (locked-down namespace), so triggers had been sending `Authorization: Bearer ` (empty) and the Edge Function rejected 100% of calls with 401. Pushes had been silently broken since at least the 2026-05-08 round.

### 2. Client: 1.0.7 hotfix bundle
Files (uncommitted on `hotfix/1.0.6`):
- [Carry/Views/GroupManagerView.swift](../Carry/Views/GroupManagerView.swift)
- [Carry/Views/GroupsListView.swift](../Carry/Views/GroupsListView.swift)
- [Carry/Views/PlayerGroupsSheet.swift](../Carry/Views/PlayerGroupsSheet.swift)
- [Carry/Views/RoundCoordinatorView.swift](../Carry/Views/RoundCoordinatorView.swift)
- [Carry/Views/ScorecardView.swift](../Carry/Views/ScorecardView.swift)

Five fixes:
1. **Index % persistence** — handicap allowance saves to server on Game Options Save, hydrates on view mount via new `initialHandicapPercentage` plumbing, refresh syncs cross-device, `handicapPercentageLastSavedAt` 8s race guard prevents 30s-poll clobber, parent groups[] notified after save so re-entry shows fresh value.
2. **"Cancel Round" → "Restart Round"** — renamed everywhere user-visible (alert title, button, menu item, in-flight text, icon `trash` → `arrow.counterclockwise`). Restart now does in-place phase transition (.active → .setup) instead of bouncing all the way out, preserving Quick Game guests + setup state. Snapshots `roundConfig.players` into `postCancelMembers` so guests added during setup also survive.
3. **Score Keeper lock follows creator** — `PlayerGroupsSheet` now takes `creatorId` parameter. Lock indicator on Score Keeper slot follows creator's actual group (not hardcoded Group 1). Binding setter rejects writes only on creator's group, allowing scorer reassignment on every other group. `syncScorerIDs` invariant uses `creatorId` not `currentUserId`.
4. **Quick Game guest preservation across refreshes** — `refreshGroupData` re-merges in-memory Quick Game guests with the server snapshot (which is Carry-only by architectural rule). Skins Groups skip preservation. Plus an `onChange(of: isQuickGame)` handler that clears the guests bucket + force-refreshes when Quick Game converts to Skins Group.
5. Build bump `CURRENT_PROJECT_VERSION` 79 → 80 (still pending).

**Status:** UNCOMMITTED, on `hotfix/1.0.6` working tree. Build bump still pending.

**A.2 All 4 push functions wired through helpers**
```sql
SELECT 
  proname AS function_name,
  pg_get_functiondef(oid) ~ '_push_notification_url\(\)' AS uses_url_helper,
  pg_get_functiondef(oid) ~ '_push_notification_anon_key\(\)' AS uses_key_helper
FROM pg_proc
WHERE proname IN (
  'notify_push',
  'send_handicap_reminders',
  'reconcile_phone_invites_for_profile',
  'reconcile_phone_invite_at_insert'
)
ORDER BY proname;
```
Expect 4 rows, all `true / true`.

**A.3 Synthetic trigger fires + returns 200**
```sql
UPDATE group_members SET status = status WHERE id IN (SELECT id FROM group_members LIMIT 1);
-- wait ~10s
SELECT date_trunc('minute', created), status_code, COUNT(*)
FROM net._http_response
WHERE created >= now() - interval '5 minutes'
GROUP BY 1, 2 ORDER BY 1 DESC;
```
Expect: `status_code = 200`. Any `401` here means Verify JWT is on but Vault read is failing.

## Failure modes to watch

- **A.3 returns 401** → Verify JWT is on but Vault read failed. Check Vault helper return value first (A.1). If helpers return empty, `vault.create_secret` may have failed silently.
- **C.1 reverts to 100%** → Persistence path not active. Check `handicapPercentage` field in the `SkinsGroupUpdate` call ([GroupManagerView.swift:1854](../Carry/Views/GroupManagerView.swift:1854)).
- **C.5 guests vanish** → Either `postCancelMembers` snapshot didn't capture, OR the 3s post-onAppear refresh re-fetched and wiped. Check for "Quick Game" branch in `refreshGroupData` ([GroupManagerView.swift:850-855](../Carry/Views/GroupManagerView.swift:850)).
- **C.7 lock on wrong group** → `creatorId` plumbing didn't reach `PlayerGroupsSheet`. Check call site at [GroupManagerView.swift:2059](../Carry/Views/GroupManagerView.swift:2059).
- **D.1 push doesn't arrive** → Could be Vault path issue (run A.1-A.3 to confirm) OR APNs cert/key mismatch (check edge function logs for "BadDeviceToken" or similar APNs errors).
